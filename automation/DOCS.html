01Overview

This suite exists to give both take-home exercises a real, repeatable regression check — not just a one-time manual pass. It uses Playwright Test with TypeScript and the Page Object Model (POM), so locators and page interactions live in one place (pages/) and the test files (tests/) read like a QA test case: arrange, act, assert.

It runs entirely against the two live public sites — no mocking, no fixtures standing in for the real app — because the goal is to catch real behavior, including the bugs documented in docs/exercise-1-the-internet/bug-board.md and docs/exercise-2-saucedemo/bug-board.md.

02Requirements

Tool Version Notes
Node.js 18 or newer Tested on 20.x
npm 9 or newer Ships with Node
Browser binary Chromium Installed automatically in step 3 below — no separate download needed

Only Chromium is required. The mobile-viewport tests use Chromium's built-in device emulation (a 375×667 viewport with touch enabled) rather than Playwright's WebKit-based devices['iPhone SE'] profile — same responsive CSS behavior, one fewer ~150MB browser binary to install on a fresh machine.

03Installation

  1. Unzip the project and move into it. 
    # if you received automation.zip
unzip automation.zip
cd automation
  2. Install dependencies. 
    npm install
  3. Install the Chromium browser Playwright drives. 
    npx playwright install chromium

That's it. No environment variables, no test accounts to provision, no local server to start — both target sites are public and already live.

04Running the tests

Command What it does
npm test Runs the full suite (both exercises, all 3 projects)
npm run test:ex1 Exercise 1 only (the-internet.herokuapp.com)
npm run test:ex2 Exercise 2 only (saucedemo.com, desktop + mobile)
npm run test:headed Same as npm test but with a visible browser window
npm run test:debug Opens the Playwright Inspector for step-through debugging
npm run report Opens the last HTML report (traces, videos, screenshots on failure)

You can also target a single file or test by name, same as any Playwright project:

npx playwright test tests/saucedemo/checkout.spec.ts
npx playwright test -g "valid credentials"

Expect exit code 0. All 52 tests report as passed, including the 17 that document currently-open bugs — see §7 for why that's intentional.

Why Page Object Model

Every locator for a screen lives in exactly one class. When the DOM changes, you fix it in one file instead of hunting through every spec that touches that screen — and it's the same reason the healer agent can repair a broken selector safely: it knows exactly where to look.

07How known bugs are handled

17 of the 52 tests document a real, reproduced bug from the bug boards rather than a passing requirement. Each one asserts the correct expected behavior and is marked with Playwright's test.fail(): 

test('TC-EX1-UPLOAD-02 [BUG-EX1-001] submitting without a file should show a validation message, not a server error', async ({ page }) => {
  test.fail(true, 'BUG-EX1-001: submitting the form with no file selected returns HTTP 500');
  const upload = new FileUploadPage(page);
  await upload.open();
  const response = await upload.submitWithoutFile();

  // Expected: a 4xx validation response (or an inline client-side error) - not a 5xx crash.
  expect(response.status()).toBeLessThan(500);
});

This keeps npm test green (the correct semantics for "regression suite you can trust in CI") while still surfacing every open bug in the HTML report — they show with a red ✘ in the list output but count toward the pass total, tagged "expected to fail."

The useful part: if a bug gets fixed upstream, that test starts passing — which Playwright reports as an unexpected pass and fails the run. That failure is the signal to delete the test.fail() line and close the bug card.

08Writing a new test

Add a method to the relevant Page Object if the interaction doesn't exist yet, then write the spec against it: 

import { test, expect } from '@playwright/test';
import { InventoryPage } from '../../pages/saucedemo/InventoryPage';

test('TC-EX2-INV-06 example: a new inventory assertion', async ({ page }) => {
  const inventory = new InventoryPage(page);
  // ...arrange, act, assert
});

Keep test IDs consistent with the scheme already in use (TC-EX<exercise>-<AREA>-<n>) and add the corresponding row to docs/exercise-*/test-cases.md so the spreadsheet and the code never drift apart.

09Formatting & pre-commit

Prettier formats the project (.prettierrc.json); lint-staged runs it only on staged files, wired into a Husky pre-commit hook. It's automatic once npm install has run (the prepare script installs the hook) — nothing else to configure. 

# runs automatically on `git commit`; to run it by hand:
npm run format         # rewrite files
npm run format:check   # CI-style check, no writes

10The healer agent Claude Code subagent

Public demo sites like these two change their markup occasionally without notice. Rather than leaving broken locators for a human to triage, this repo ships a Claude Code subagent at .claude/agents/healer.md whose only job is fixing that specific failure mode.

What it does

  1. Reproduces the failing test and reads the Playwright trace/error to find the broken locator.
  2. Inspects the live DOM at the point of failure to find the element's new shape.
  3. Proposes a robust replacement locator (role/text/testid, per Playwright's own locator priority) and patches the one Page Object file responsible — never the spec files, never unrelated pages.
  4. Re-runs just that test to confirm the fix, and reports a short diff.

When to reach for it

Run npm test. If a failure isn't one of the 17 documented bugs and looks like a locator problem (a timeout waiting for a selector, not an assertion mismatch), invoke the agent from Claude Code: 

› @healer the login test in automation/tests/saucedemo/login.spec.ts is failing, 
  looks like a broken locator, can you take a look?

It intentionally will not touch test assertions or bug annotations — those are QA judgment calls, not something to auto-resolve.

11Troubleshooting

"Executable doesn't exist" when running tests
Chromium isn't installed yet — run npx playwright install chromium.

A saucedemo test times out on #login-button or similar
The site occasionally rate-limits rapid repeated runs from the same IP. Wait a minute and re-run, or run with --workers=1 to reduce concurrent sessions.

A test fails that isn't one of the 17 documented bugs
Check automation/playwright-report/index.html (npm run report) for the trace. If it's a changed selector, see §10. If it's a new, real behavior change on the target site, add it to the relevant bug-board.md and wrap the assertion in test.fail() following the pattern in §7.

12Continuous Integration

.github/workflows/e2e-tests.yml runs this suite in GitHub Actions on every push and pull request to main, and on demand via the Actions tab (workflow_dispatch). It mirrors the two npm scripts from §04 as two independent jobs:

Job Command Playwright project(s)
exercise-1 npm run test:ex1 the-internet-desktop
exercise-2 npm run test:ex2 saucedemo-desktop, saucedemo-mobile

Both jobs run in parallel on ubuntu-latest: checkout → Node 20 (with npm cache keyed on automation/package-lock.json) → npm ci → npx playwright install --with-deps chromium → the test script. Each job uploads automation/playwright-report/ as a build artifact (if: always(), 14-day retention) whether it passes or fails, so a red run can be triaged straight from the Actions tab — download the artifact and open index.html to see traces, videos, and screenshots, no local reproduction needed first.

A green run is the expected result, not just "no failures." As covered in §7, 17 of the 52 tests intentionally assert the correct behavior for a known, open bug and are marked with test.fail() — they show up in the job log with a red ✘ next to their name, but count toward the pass total and don't fail the build. Only an actual unexpected pass (one of those 17 starts succeeding) or a genuinely new failure will turn a run red.

Two concurrency safeguards keep the workflow well-behaved on a fast-moving branch:

concurrency:
  group: e2e-${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

A new push to the same branch cancels any run already in progress for it, so CI minutes aren't spent finishing a run that's already obsolete.

Running it locally the same way CI does

# from automation/, same commands the workflow runs
npm ci
npx playwright install --with-deps chromium
npm run test:ex1
npm run test:ex2

playwright.config.ts already detects process.env.CI and reduces parallel workers accordingly, so behavior between a local run and the Actions run stays consistent.
